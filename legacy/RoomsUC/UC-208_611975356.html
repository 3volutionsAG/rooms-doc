Name

Reservieren von Personen

Ziel

Analog eines Equipment soll eine Person mit reserviert werden können

Abhängigkeiten zu anderen UC

101, 203, 206, 107, 109



Wichtige Anmerkungen

Jede Liste (Grid) beinhaltet immer das Personalisieren von Spalten, Sortierung und Paging sowie optionale Checkboxen für Mehrfachselektion.

Jede Detailansicht beinhaltet immer einen Readonly-Modus, einen Editier-Modus für Einzelmutationen und einen Editier-Modus für Mehrfachmutationen.

Diese Funktionalitäten müssen deshalb beim Beschreiben der Use Cases nicht aufgeführt werden.

Zudem müssen Validierungsregeln zum aktuellen Zeitpunkt nicht beschrieben werden.

Grobbeschrieb

Bitte hier den Use Case im groben Beschreiben – auch Abhängigkeiten zu anderen Use Cases sind zu erwähnen:

Innerhalb einer Raumreservation, sowie auch separat soll eine verfügbare Ressource (z.B. Chauffeur) reserviert werden können.

  • Suche nach einer Ressource innerhalb und ausserhalb einer Objektreservation unter Verwendung verschiedener Suchkriterien (verfügbar/alle, Kategorie, Datum, Zeit)
  • Anzeige der verfügbaren Ressourcen in Form eines Kalenders (Free/Busy Ansicht) unter Berücksichtigung der Berechtigungen.
  • Selektion einer oder mehrere Ressourcen
  • Reservation der Ressourcen – allenfalls Verlinkung mit der Parent-Reservation (Objekt)

Subcases und Features

Bitte auflisten und im groben beschreiben:

  • Konfiguration: Welche Objektetypen unterstützten Ressourcenreservation
  • Innerhalb der „Verwaltung von Personen (UC101)“ muss eine Person als reservierbar markiert werden können.
  • Einer reservierbaren Person muss ein Standort und Reservationskategorien (Skills) zugewiesen werden können (Lehrer, Chauffeur, Dienstleister, etc.)
  • Eine Ressource kann innerhalb seines Kalenders Sperrzeiten (Abwesenheiten) definieren.
  • Ressourcenreservation muss mit Outlook synchronisiert werden können
  • Mail/SMS Notifikation muss konfiguriert werden können
  • Definition der Berechtigung/Benutzergruppe auf einer Ressource
